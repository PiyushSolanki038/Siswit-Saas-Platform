Overall Architectural Recommendations
Based on patterns observed across all 35 issues in the audit, here are 10 high-
level recommendations:
1. Unify Data Scoping into a Single Pattern
The codebase currently has three scoping approaches: applyModuleReadScope
(CRM/CPQ/CLM/ERP), applyTenantOwnershipScope (Documents), and
raw .eq("organization_id", ...) (AuthProvider). Standardize on
module-scope.ts for all modules, and create an admin-scope helper for
AuthProvider-level operations.
28


===== PAGE 29 =====
2. Add organization_id to Every T able
10 child tables rely on parent JOINs for RLS. This is fragile, slower, and
error-prone. Every table that stores business data should have its own
organization_id NOT NULL column with a sync_scope_ids trigger.
3. Centralize Error Handling
getErrorMessage is duplicated 6× and growing. Create src/core/utils/errors.ts
with a single export. Add an ESLint rule to prevent local re-declarations. Also
replace console.error with a pluggable logger.
4. Eliminate All as any and as unknown as Casts
There are 15+ type-bypass casts across the codebase. Create a typed Supabase
client wrapper ( typedFrom<T>) and make buildModuleCreatePayload fully
generic. Add noExplicitAny: true to tsconfig.json and enforce in CI.
5. Standardize Mapper F unctions
Every map* function in the hooks is hand-written and error-prone (B-01, B-
02, B-03 are all mapper bugs). Consider generating mappers from a schema
definition, or at minimum create a base mapper test that verifies every frontend
field maps to a real DB column.
6. Enforce Soft-Delete Consistency
Some operations hard-delete child rows (B-04) while soft-deleting parents. Cre-
ate a cascadeSoftDelete(parentTable, parentId, childTables[]) utility
and use it everywhere. Add a BEFORE DELETE trigger that prevents hard deletes
on business data tables.
7. Add Integration T ests for Multi-T enancy
None of the cross-tenant attack vectors (C-01 through C-07) would survive a
test that logs in as Tenant A and tries to read/write Tenant B data. Add a
Playwright or Vitest integration test suite that specifically tests tenant isolation
boundaries.
8. Consolidate Realtime Subscriptions
Documents creates 3+ WebSocket channels simultaneously. Create a single
RealtimeProvider per module (or per app) that multiplexes all subscriptions
through one channel, reducing connection overhead and simplifying cleanup.
29


===== PAGE 30 =====
9. Move Business Logic to the Database Layer
Client-side quote total recalculation (P-03) and status transitions (B-05) are
susceptible to race conditions and bypass. Move these to database triggers or
RPCs where atomicity is guaranteed.
10. Adopt a Code Generation Strategy
Duplicate types (Q-03), duplicate scope hooks (Q-04), and mapper drift (B-
01, B-02, B-03) all stem from manual copy-paste. Invest in codegen: - Auto-
generate frontend types from the Supabase Database type - Auto-generate map-
per functions from a type mapping config - Auto-generate CRUD hooks from a
per-table config file
